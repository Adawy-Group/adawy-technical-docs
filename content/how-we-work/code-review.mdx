# Code Review

Every change is reviewed before it merges. Today the **lead & technical
owner** reviews all PRs; as the team grows, review becomes peer-based with
the lead reviewing platform-level changes.

## What reviewers check, in order

1. **Correctness** — does it do what the card says? Edge cases, error states,
   loading states.
2. **Platform impact** — a change in `@adawy/*` packages ships to *every*
   brand app: does it behave for all of them, or only the one being worked
   on? Conversely, is reusable code being forked into an app when it belongs
   upstream in `packages/`
   ([keep apps thin](/standards/project-structure))?
3. **Standards** — [coding conventions](/standards/coding), accessibility,
   bilingual/RTL readiness, performance budget (no regression on Lighthouse).
4. **Tests and checks** — automated checks green, and changed behavior covered
   where it matters.
5. **Clarity** — will the next person understand this without asking the
   author?

## Expectations

- **Authors:** keep PRs small, link the card, describe *why* not just *what*,
  and self-review the diff before requesting review. The preview link should
  show the change working.
- **Reviewers:** respond within one working day. Review the code, not the
  person. A disagreement is resolved with a reason ("this breaks brand
  theming because…"), not a preference.
- **Approval is required** before merge — work is only *ready to build* once
  its design is approved and fully specified, and only *done* once it has
  passed review and staging checks (see
  [Definition of Done](/how-we-work/definition-of-done)).
