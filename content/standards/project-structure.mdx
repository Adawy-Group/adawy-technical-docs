# Project Structure

## The platform monorepo

The platform is **one monorepo (Turborepo, pnpm workspaces)**: a thin Next.js
app per flagship brand in `apps/`, built entirely from shared `@adawy/*`
packages in `packages/`. Each flagship app deploys to its own domain as its
own Vercel project; the long tail of templated brands shares one multi-tenant
app ([ADR-008](/architecture/adrs)).

```
adawy-platform/
├── apps/
│   ├── group/      Adawy Group   → adawygroup.com   (flagship — own app; the only app today)
│   ├── studio/     Adawy Studio  → adawystudio.com  (flagship — own app, planned)
│   ├── pack/       Adawy Pack    → adawypack.com    (flagship — own app, planned)
│   ├── sites/      Long-tail brands — one multi-tenant app, hostname → config (planned, ADR-008)
│   └── api/        NestJS backend (Phase 2, deployed off-Vercel — ADR-009)
└── packages/
    ├── ui/                 @adawy/ui                (shadcn primitives + sections)
    ├── design/             @adawy/design            (Tailwind v4 token structure, CSS-first)
    ├── motion/             @adawy/motion            (GSAP + Lenis primitives)
    ├── i18n/               @adawy/i18n              (next-intl routing, RTL helpers)
    ├── eslint-config/      @adawy/eslint-config     (architecture-enforcing lint rules)
    ├── typescript-config/  @adawy/typescript-config (shared tsconfig bases)
    ├── seo/                @adawy/seo               (metadata + sitemap helpers — planned)
    └── utils/              @adawy/utils             (shared utilities — planned)
```

> **Today** only `apps/group` and the packages it uses (`ui`, `design`,
> `motion`, `i18n`, `eslint-config`, `typescript-config`) exist. The other apps
> and the `seo`/`utils` packages are added as the platform grows — extraction
> is deliberate, on the third use (rule 5 below), not up front.

A new brand is a **config entry in `apps/sites`** by default. It gets its own
folder under `apps/` only when it needs unique routes, custom sections, or
bespoke motion — the decision rule lives in the
[launch guide](/guides/launch-new-brand). Packages ship as TypeScript source
(no build step); apps compile them.

## The rules behind the layout

1. **Apps are thin; packages own the logic.** An app contributes its routes,
   its content, and its token values — the components, motion patterns, SEO
   helpers, and utilities it composes all come from `@adawy/*`. If you are
   writing a reusable component inside an app, stop: it belongs in
   `packages/ui`, themed by tokens. *Why:* this is what makes a fix ship once
   and reach every brand ([ADR-001](/architecture/adrs)).
2. **Where code goes:**
   - Visual primitives and composed sections → `@adawy/ui`
   - Design tokens, theme structure, Tailwind base → `@adawy/design`
   - Animation primitives and scroll machinery → `@adawy/motion`
   - Locale routing, direction handling, next-intl config → `@adawy/i18n`
   - Metadata, Open Graph, sitemap/robots helpers → `@adawy/seo`
   - Everything shared and unglamorous → `@adawy/utils`
3. **Brand identity lives in tokens, not forks.** A brand customizes by
   overriding token values from `@adawy/design` — never by copying a
   shared component into its app and editing it. Genuinely custom sections are
   for flagships only and are explicitly scoped.
4. **Shared means machinery, not appearance.** Fonts, colors, SEO content,
   and motion *choreography* are per-brand by design: `@adawy/design` defines
   the token slots (`--brand-font-display`, `--brand`, …) and each brand
   assigns its own values; `@adawy/seo` is plumbing fed with brand content;
   `@adawy/motion` ships primitives that brands compose into their own feel.
   The system exists to make brands siblings, not clones.
5. **Generalize on the third use, not the first.** A section used by one app
   can live in that app; the moment a second or third brand needs it, it moves
   to `packages/` — extraction is a deliberate step, not a default.
6. **Turborepo runs the builds.** Only affected apps rebuild on a change; a
   PR that touches `@adawy/ui` shows every affected app's preview, because
   that change ships to all of them.

## This documentation site

```
adawy-technical-docs/
├─ app/                  # Nextra app shell ([[...mdxPath]] catch-all + layout)
├─ content/              # All handbook pages (MDX) — sidebar mirrors folders
│  └─ <section>/_meta.ts # Sidebar order & titles per section
├─ next.config.mjs       # Nextra config
└─ mdx-components.tsx
```

To add a page: create the `.mdx` file in the right section folder, add it to
that folder's `_meta.ts`, open a PR.
