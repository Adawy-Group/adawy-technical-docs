# Coding Conventions

Conventions exist so anyone can work in any repo. Each rule states its reason;
a rule without a reason is a candidate for deletion.

## TypeScript

- **Strict mode, always.** `strict: true` in every tsconfig. *Why:* the
  platform serves 38 brands from one codebase — a type error caught at build
  time is a bug prevented on 38 sites at once.
- **No `any` in committed code.** Use `unknown` + narrowing, or fix the type.
  *Why:* `any` silently disables the protection we pay for.
- **Validate at the boundary with Zod.** Every external input — form
  submissions, API responses, Odoo webhooks, CMS content — is parsed with a
  Zod schema before it enters typed code. *Why:* types are compile-time only;
  the boundary is where runtime reality enters.
- **TSDoc on every exported symbol.** Package entry points carry
  `@packageDocumentation`; exports document their role and the rule they
  enforce, not a restatement of the signature. *Why:* the packages are the
  product — a brand developer should understand `@adawy/*` from the editor
  tooltip, without reading the source.

## Next.js / React

- **Server Components by default; `"use client"` only where interaction
  requires it.** *Why:* this is a core part of how we hold the Lighthouse 94
  target — less JavaScript shipped is the cheapest performance win.
- **`next/image` for every image; no raw `<img>`.** *Why:* WebP/AVIF and
  responsive sizing cut 25–50% of image weight — the exact failure of the old
  sites.
- **Brand theming flows through CSS variables (design tokens), never
  hard-coded values.** *Why:* a hard-coded color works on one brand and breaks
  on 37 others.
- **Forms: React Hook Form + Zod resolver.** One validation schema serves
  client and server.

## Naming & files

The trap is conflating the **file name** with the **symbol it exports** — they
use different cases. Hold them apart and the rules never collide.

**Files and folders are always `kebab-case`** — components, hooks, utilities,
routes, and assets alike: `hero-intro.tsx`, `use-media-query.ts`,
`structured-data.ts`. *Why:* one rule for every file ends the daily "which case
was this?" lookup, matches both the shadcn primitives we vendor (`button.tsx`)
and Next.js's own route files (`page.tsx`, `layout.tsx`), and sidesteps the
case-sensitivity bugs that surface when case-insensitive macOS meets
case-sensitive Linux CI.

**The exported symbol keeps its idiomatic case** — the file is kebab, the thing
inside it is not:

| Symbol | Case | Example (`file` → `export`) |
| ------ | ---- | --------------------------- |
| Component / provider | `PascalCase` | `theme-toggle.tsx` → `ThemeToggle` |
| Hook | `useCamelCase` | `use-media-query.ts` → `useMediaQuery` |
| Function | `camelCase`, verb-first | `pwa.ts` → `buildManifest` |
| Boolean / predicate | `is` / `has` / `should` + `camelCase` | `isStandalone`, `hasSeenPreloader` |
| Type / interface (props end in `Props`) | `PascalCase` | `Locale`, `ThemeToggleProps` |
| Fixed-scalar constant | `SCREAMING_SNAKE_CASE` | `PRELOADER_SEEN_KEY`, `SNOOZE_DAYS` |

**Design tokens and the utilities that read them are `kebab-case`:** brands
override the slots `--brand`, `--brand-foreground`, `--brand-font-display`, and
`--brand-font-body`, consumed as `bg-brand`, `font-heading`, `font-sans`.
*Why:* the token name is the brand API — a stable kebab slot is what lets a
single override repaint a brand ([project structure](/standards/project-structure)).

**i18n casing:** locale codes and message files are lowercase (`ar`, `en.json`);
message namespaces are `PascalCase` (`HomePage`, `Metadata`); leaf keys are
`camelCase` (`installTitle`).

- One component per file; co-locate component-specific helpers and types with
  the component.
- Branches and commits follow the [git workflow](/how-we-work/git-workflow):
  `feat/`, `fix/`, `content/`, `chore/` — mirroring the board's Type labels.

## Quality bars (enforced, not aspirational)

- **Accessibility:** WCAG 2.1 AA. Semantic HTML first, ARIA second. Every
  image has meaningful alt text (or `alt=""` if decorative); decorative
  marquee duplicates get `aria-hidden="true"`. *Why:* these were audit
  findings on the old site — we do not re-introduce documented mistakes.
- **Performance:** Lighthouse mobile ≥ 90 on every page; heavy media re-encoded
  (background video ≤1080p, poster frame, `preload="none"`,
  `prefers-reduced-motion` respected). *Why:* our audience loads pages on
  Egyptian 4G; every megabyte is real money and real abandonment.
- **Bilingual-ready:** no concatenated strings that break under translation;
  layouts must survive RTL. *Why:* Arabic is not an afterthought — it is the
  market.
- **SEO:** every page sets a real title (50–60 chars), meta description
  (140–160 chars), Open Graph tags, and structured data where applicable.
  *Why:* the old site's tab literally said "About Page – Adawy Group".

## Linting & formatting

ESLint + Prettier run in CI; `@adawy/eslint-config` in the platform repo is
the source of truth. Don't argue style in review — the formatter wins.

The lint philosophy: **a rule is an error or it is not enabled.** No
warnings — they scroll past and never get fixed, and with one developer per
brand a warning-only lint is no lint at all.

Beyond the standard presets (type-aware `typescript-eslint`, React + hooks,
`jsx-a11y`, Next.js Core Web Vitals), the config machine-enforces the
architecture — the rules in this handbook fail the build, not just review:

| Enforced rule | What it guards |
| ------------- | -------------- |
| `gsap`/`lenis` imports banned outside `@adawy/motion` | Single plugin registration, SSR safety, one shared ticker ([project structure](/standards/project-structure)) |
| `next/link` and `next/navigation`'s router APIs banned in apps — use `@/i18n/navigation` | Every URL carries the correct locale prefix |
| Physical Tailwind classes (`ml-`/`mr-`/`pl-`/`pr-`/`left-`/`right-`/`text-left`/`text-right`) are errors — use logical ones (`ms-`/`me-`/`ps-`/`pe-`/`start-`/`end-`/`text-start`/`text-end`) | Layouts mirror correctly in Arabic (RTL) without retrofitting |
| `no-floating-promises` (type-aware) | An unawaited promise in a Server Component is a silently dropped error |
