# Our Systems

## The unified platform (what we are building)

| System | What it is | Status |
| ------ | ---------- | ------ |
| **Brand sites platform** | One Turborepo monorepo: a thin Next.js app per flagship brand (`apps/group`, `apps/studio`, `apps/pack`) plus one multi-tenant `apps/sites` for the long tail ([ADR-008](/architecture/adrs)), all on shared `@adawy/*` packages, each app deployed to its own domain(s) | Phase 1 — in build |
| **Shared design system** | The `@adawy/*` packages: `ui` (shadcn primitives + sections), `design` (Tailwind v4 design tokens), `motion` (GSAP + Lenis), `i18n` (next-intl + RTL), `seo`, `utils` — themeable per brand via tokens | Phase 1 — the most important technical deliverable |
| **Backend service** | Node.js (NestJS) in the same monorepo as `apps/api` ([ADR-009](/architecture/adrs)): authentication, business logic, Odoo integration, background jobs, real-time | Phase 2 — first build of the phase |
| **Aladawy Shop (rebuild)** | Custom e-commerce on the same stack; catalog synced from Odoo; Paymob (cards) + Fawry (cash on delivery) | Phase 2 |
| **Client portal** | portal.adawygroup.com — orders across all sub-brands, brand assets, invoices, structured intake, AI assistant | Phase 2 — the flagship |
| **Employee portal** | One internal home: internal AI assistant, search across all company systems, operational dashboards, content engine | Phase 3 |
| **AI solutions** | Member-facing assistants and lead scoring (Phase 2); internal assistant, search, content engine, manufacturing intelligence (Phase 3) | Phases 2–3 |

## What runs where

| On Vercel (frontend) | On the backend host (Railway → DigitalOcean) |
| -------------------- | -------------------------------------------- |
| All brand websites — one Vercel project per app | Authentication and sessions |
| SSR and static generation | Business logic and Odoo integration |
| Client portal UI | Background jobs and scheduled tasks |
| Image optimization | Real-time connections (Pusher) for the portal |
| Preview deployments per PR (affected apps only) | Webhooks from Odoo |

The split exists because Vercel functions can't hold persistent connections
and time out on long-running jobs — fine for serving pages, wrong for Odoo
webhooks and queues. PostgreSQL runs managed (Supabase or Neon); Redis on
Upstash.

## Odoo — the system of record

The group runs on **Odoo** for accounting, payroll, inventory, supply chain,
HR, and manufacturing. **We keep it permanently** and build custom layers on
top through its APIs. Commodity functions (payroll, tax compliance) are not
where we differentiate; client experience, dashboards, and AI are.

Flow example: a client orders on Aladawy Shop → the backend writes the order
to Odoo → Odoo handles inventory, accounting, fulfilment → the client sees
status in the portal, surfaced from Odoo via our cache. Odoo's current API is
deprecated in Odoo 22 (late 2028), so the integration sits behind an
abstraction layer — migrating is one or two sprints, not a rewrite.

## The legacy systems (what we are replacing, and why)

| Site | Old stack | Verdict from the technical audit |
| ---- | --------- | -------------------------------- |
| adawygroup.com | WordPress 6.9 + Semplice page builder | Lighthouse mobile ~32; 4K background videos; stock photography; CMS-default page titles; no meta/OG tags; no structured data; English only. 20 documented findings. |
| aladawypack.com | WordPress | Separate theme/plugins, no shared code |
| aladawyshop.com | WordPress + WooCommerce + Woodmart | Demo content still visible; stack never designed for multi-brand commerce |

The audit's conclusion — accepted by leadership — was that these are problems
of **platform fit, not execution**: no amount of WordPress optimization makes
a page-builder deliver the cinematic, bilingual, brand-driven experience the
group's positioning requires. Hence the rebuild ([ADR-002](/architecture/adrs)).
