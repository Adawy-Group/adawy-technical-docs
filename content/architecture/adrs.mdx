# Architecture Decision Records

Significant technical decisions, recorded so future team members understand
not just *what* we decided but *why* — and what we knew at the time. Format:
**Context → Decision → Consequences.** New ADRs are added via PR.

## ADR-001 — One monorepo: a thin app per brand on shared packages

- **Context:** 38 domains today, 110 planned. Fully separate sites mean every
  change, patch, and feature ×38 — roughly 30 engineers at maturity. A single
  multi-tenant app with domain routing was considered, but it couples every
  brand to one deployment and makes flagship-level customization awkward.
- **Decision:** One Turborepo monorepo. Each brand is a **thin Next.js app**
  (`apps/group`, `apps/studio`, `apps/pack`, …) deployed to its own domain as
  its own Vercel project. All real engineering lives in shared packages:
  `@adawy/ui` (shadcn primitives), `@adawy/tailwind-config` (design tokens),
  `@adawy/motion` (GSAP), `@adawy/seo`, `@adawy/utils`.
- **Consequences:** A fix in a shared package ships to every brand from one
  PR; Turborepo rebuilds only affected apps. Each brand deploys independently —
  a bad deploy on one site cannot take down the others, and flagships keep
  full creative freedom in their own app. Trade-offs: a shared-package change
  re-deploys every affected app, and discipline is required to keep apps thin —
  reusable code must flow upstream into `packages/`, never get forked into an
  app ([project structure](/standards/project-structure)).

## ADR-002 — Rebuild on a code-first stack instead of patching WordPress

- **Context:** The 2026 technical audit of adawygroup.com documented 20
  findings (Lighthouse mobile ~32, template footer credits, broken sub-brand
  links, missing SEO/meta/structured data, stock photography, English only).
  Root cause: page-builder platform fit, not execution.
- **Decision:** Rebuild from scratch on Next.js, accepted by
  leadership. WordPress is right for a small business with simple needs and
  wrong for a 38-company group operating one coordinated digital experience.
- **Consequences:** Full code/content ownership, real version control and
  review, bilingual/RTL by design, cinematic motion possible. Cost: we build
  and own everything — hence the design system investment in Phase 1.

## ADR-003 — Keep Odoo as the system of record

- **Context:** The group runs finance, inventory, payroll, HR, and
  manufacturing on Odoo. Custom ERPs almost always end in regret.
- **Decision:** Keep Odoo permanently. Build differentiating layers (portal,
  dashboards, AI) on top via its APIs, behind an abstraction layer; cache in
  our PostgreSQL.
- **Consequences:** Years of engineering saved. Odoo API deprecation (Odoo 22,
  late 2028) costs one or two sprints thanks to the abstraction. SAP is a
  Phase-4 evaluation, not a today problem — and the platform doesn't depend on
  Odoo specifically.

## ADR-004 — Defer the CMS; content is managed in the repo for now

- **Context:** A unified admin dashboard (originally planned on Payload CMS)
  was considered for Phase 1 so non-developers could edit every brand's
  content. With a two-person team and the website launch as the priority,
  building and operating a CMS now adds scope without near-term benefit.
- **Decision:** Defer the CMS. Site content lives in the repository (MDX and
  configuration), edited by the team and reviewed via PR like any other
  change. Revisit when content volume or non-developer editing demands it —
  and if adopted, it will be self-hosted, never a per-seat hosted service.
- **Consequences:** Zero CMS infrastructure to run today; every content change
  is versioned and reviewed. Trade-off: content edits require a developer in
  the loop until a CMS decision is made.

## ADR-005 — Custom e-commerce instead of Shopify/WooCommerce

- **Context:** Aladawy Shop runs on WooCommerce; the group may eventually sell
  across multiple storefronts sharing one inventory, one customer base, one
  Odoo integration. Shopify means a subscription + transaction fees per store.
- **Decision:** Custom commerce on the platform stack — catalog in PostgreSQL
  synced from Odoo, Paymob (cards) + Fawry (cash on delivery).
- **Consequences:** No per-store fees or vendor lock-in; storefronts for new
  brands (or ecosystem members, Phase 3–4) ship in days. Cost: we own cart,
  checkout, and payment-integration code — using well-established Next.js
  commerce patterns, not from scratch.

## ADR-006 — Every brand keeps its own domain, cross-linked to the group

- **Context:** Three options: subdomains of adawygroup.com, fully separate
  domains, or own domains with cross-linking.
- **Decision:** Own domains with "part of Adawy Group" cross-links both ways
  (the LVMH model).
- **Consequences:** Each company reads as a real company, not a corporate
  department; the ecosystem stays discoverable. Each domain points at its
  brand app's own Vercel project (Cloudflare in front).

## ADR-007 — Documentation lives in this repo, public, docs-as-code

- **Context:** Department knowledge was scattered across chats and PDFs; new
  members had no single source of truth.
- **Decision:** This Nextra site, public, on the same Vercel/GitHub workflow
  as everything else. Changes via PR.
- **Consequences:** Onboarding is self-serve; docs rot is visible in git; the
  handbook itself demonstrates the platform's standards.
